In our last post, we covered Progressive Hydration. Today, we'll focus on Streaming Server-Side Rendering, an innovative approach that streams HTML content from the server to the client as it is being rendered. This technique reduces time-to-first-byte and improves perceived performance by allowing users to see content faster.

We think this conversational style might be a bit more fun and easier to follow than a regular article. Hopefully, it makes learning about these rendering patterns less boring and more hands-on!

<Conversation.Thread from="server">
  Of course! Streaming Server-Side Rendering (SSR) is a technique
  where the server starts sending HTML content to the client as soon
  as it's available, instead of waiting for the entire page to be
  rendered before sending anything. This can significantly improve the
  time to first byte (TTFB) and the overall perceived performance of
  the page.
</Conversation.Thread>

<Conversation.Thread from="user">
  Can you give me an example of how Streaming SSR works in practice?
</Conversation.Thread>

<Conversation.Thread from="server">
  Sure! Imagine we're on a product detail page of an e-commerce site.
  This page has three main sections: Product Details, Reviews, and
  Related Products. Each section requires data from the database.
  Let's say it takes 250ms to get the Product Details, 500ms for the
  Reviews, and 400ms for the Related Products.
</Conversation.Thread>

<Conversation.Thread from="server">
  With Streaming SSR, as soon as the server gets the Product Details
  data, it starts sending that HTML to the client immediately. It
  doesn't wait for the Reviews and Related Products data to be ready.
  When the Reviews data is ready, it sends that next, and then finally
  sends the Related Products data. This way, the client can start
  displaying the page content much faster, even if some parts are
  still loading.
</Conversation.Thread>

  feel like it's loading faster and gives users something to interact
  with sooner.
